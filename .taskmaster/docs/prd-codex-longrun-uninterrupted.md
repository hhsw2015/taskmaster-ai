# Codex Longrun 无中断执行增强需求

## 背景

当前项目已经具备 `task-master codex run` 的 runner-controlled 长时执行能力：

- `Taskmaster` 负责任务图、依赖、状态回写、checkpoint、日志和重试控制。
- `Codex CLI` 负责单次任务实现，并通过 `TM_RESULT` 将结果交回 runner。
- runner 在每个任务结束后执行状态写回，再决定是否进入下一个任务。

这条链路在工程上是稳定的，但当前用户体验仍有一个明显问题：

- 用户在 `Codex CLI` 对话中说“开始实现”后，并不总能稳定进入 `task-master codex run`。
- 一旦没有真正进入 runner 自动链路，Codex 就会退回“普通对话式做任务”，做完一个任务后停下来等待用户确认。
- 结果是：系统具备“连续执行”的能力，但入口体验仍然不可靠，无法保证“一次启动后把所有可执行任务连续跑完”。

## 目标

在保留现有 runner-controlled 架构的前提下，增强 Codex longrun 的入口与执行体验，使用户在项目内与 Codex 对话时，可以稳定触发“无中断长时执行”模式，并连续执行所有可运行任务，直到：

- `all_complete`
- `blocked`
- `error`

其中“无中断”指的是：

- 不在每个任务后询问用户是否继续
- 每个任务完成后自动回写 Taskmaster 状态
- 回写成功后自动进入下一个任务
- 整个过程在前台可见，终端持续输出执行过程

## 非目标

以下内容不属于本次需求范围：

1. 不替换现有 runner-controlled 架构为纯 model-managed 模式。
2. 不改造 `loop` / `autopilot` 现有能力。
3. 不引入云端依赖；该能力必须支持纯本地执行。
4. 不要求把多个任务塞进一次 `codex exec` 中执行。
5. 不要求把 `Taskmaster` 的业务任务图改造成“任务 + 同步子任务”模型。

## 现状问题

### 问题 1：对话触发不稳定

`task-master codex init` 会把“当用户说开始实现时，应运行 task-master codex run”写进 AGENTS/Skill 资产，但这仍属于提示词约定，而不是强约束入口。

造成的结果是：

- 有时 Codex 会正确拉起 runner
- 有时 Codex 会直接按普通聊天方式实现当前任务
- 用户无法确定当前是否进入了“连续执行模式”

### 问题 2：用户无法快速判断当前执行模式

当执行链路没有进入 runner 时，用户通常只能在做完一个任务后，才发现系统停下来等确认。

这意味着缺少一个明确、可观察的执行模式信号，无法在开始阶段就判断：

- 是否已经进入 longrun runner
- 当前使用的是哪个 tag
- 当前任务是否会自动推进到下一个任务

### 问题 3：项目内默认体验不够强

当前系统已经具备稳定的每任务独立执行、状态回写、重试和 checkpoint 能力，但项目内默认入口还不够强，导致“功能已有、体验不稳定”。

## 核心设计要求

### 要求 1：现有 runner-controlled 模式保持为主路径

本次增强必须建立在当前 `task-master codex run` 模式之上，而不是引入新的默认执行核心。

系统必须继续保持：

- 一次 `codex exec` 只处理一个 Taskmaster 任务
- runner 是唯一状态写回者
- 成功 / 失败 / blocked 的判定继续由 runner 决定
- checkpoint、ledger、日志和超时控制继续保留

### 要求 2：项目内必须提供一个更强的“开始实现”入口

初始化后的项目，在 Codex CLI 对话中触发“开始实现”时，必须更稳定地进入 `task-master codex run` 主链路，而不是退回普通聊天模式。

这里的“更稳定”至少包括：

1. 有明确的项目内入口约定
2. 入口文案和 AGENTS/Skill 资产一致
3. 当前激活 tag 会被正确使用，而不是硬编码 `master`
4. 一旦进入 runner，默认连续执行直到终态，不逐任务询问

### 要求 3：开始阶段必须输出明确的执行模式信号

一旦 longrun 开始，终端必须明确告诉用户：

- 已经进入 longrun runner
- 使用的 tag
- 使用的 executor/model/reasoning 配置
- 是否会自动继续到下一个任务

这样用户在任务 1 开始之前就能知道：

- 当前是不是“真正的连续执行”
- 不是等到做完一个任务后才发现跑偏

### 要求 4：任务之间必须自动推进

当一个任务完成后，系统必须执行如下顺序：

1. 解析 `TM_RESULT`
2. 根据结果回写 Taskmaster 状态
3. 更新 checkpoint / ledger / runtime 产物
4. 如果状态写回成功且没有命中终止条件，则自动进入下一个任务

系统不得在单任务成功后询问用户“是否继续”。

### 要求 5：必须保留失败和阻塞边界

“连续执行”不能以牺牲安全边界为代价。

系统必须继续支持：

- 每任务重试上限
- 超时控制
- `blocked` 状态
- `continueOnFailure=false` 时的快速停止
- 中断后从 checkpoint 恢复

### 要求 6：前台执行过程必须可见

系统应默认在当前终端前台输出：

- 当前开始执行的任务
- 关键生命周期日志
- 每个任务的结束状态、耗时、日志文件
- 最终总结

用户不应被迫依赖后台日志文件才能知道系统是否仍在工作。

## 用户故事

### 用户故事 1

作为已经在 `Codex CLI` 对话中的用户，
我希望在完成任务拆分后，只说一句“开始实现”，
系统就稳定进入 longrun 模式并连续执行所有可运行任务，
而不是做完一个任务后停下来等我确认。

### 用户故事 2

作为需要信任自动执行的用户，
我希望在开始阶段就能看到明确的 runner 启动信号，
以便确认当前不是普通聊天模式，而是受 Taskmaster 控制的连续执行模式。

### 用户故事 3

作为需要在失败后恢复执行的用户，
我希望当某个任务失败、阻塞或中断时，
系统保留正确的状态、日志和 checkpoint，
让我可以重新启动后继续，而不是从头开始。

## 验收标准

### 验收标准 1：入口正确

在已完成 `task-master codex init` 的项目中，用户通过 Codex 对话触发“开始实现”后，应稳定进入 `task-master codex run` 链路，而不是进入普通对话式单任务实现。

### 验收标准 2：自动推进

至少使用一个包含 3 个可顺序执行任务的本地测试项目进行验证：

1. 任务 1 完成后自动回写状态
2. 无需用户确认，自动开始任务 2
3. 任务 2 完成后自动开始任务 3
4. 最终输出 `all_complete` 或等价终态总结

### 验收标准 3：可观察性

前台终端输出必须包含：

- longrun 启动提示
- 每个任务的开始日志
- 每个任务的结束日志
- 最终总结

### 验收标准 4：失败边界

在任务失败场景下，系统必须满足：

- 当未达到重试上限时，任务回到可重试状态
- 达到重试上限后写为 `blocked`
- 若配置 `continueOnFailure=false`，则执行链终止

### 验收标准 5：恢复能力

在执行过程中人为中断后，重新启动 longrun，应能够基于 checkpoint / Taskmaster 当前状态继续，而不是重复执行已完成任务。

## 建议实现方向

本次增强优先沿以下方向推进：

1. 强化项目内的 longrun 触发入口，而不是重写执行核心。
2. 增强 runner 启动时的模式标识和前台可观察性。
3. 明确“普通聊天模式”和“runner 连续执行模式”的行为分界。
4. 用真实本地项目做端到端回归，验证“3 个任务连续自动完成”的主场景。

## 成功标准

若本次需求完成，用户应能在一个已初始化的项目里达到如下体验：

1. 与 Codex 对话完成需求拆分
2. 说“开始实现”
3. 系统在当前终端明确显示已进入 longrun runner
4. 任务一个接一个自动执行
5. 每个任务完成后自动回写状态
6. 不在中间逐任务请求继续确认
7. 直到 `all_complete`、`blocked` 或 `error` 才停下
